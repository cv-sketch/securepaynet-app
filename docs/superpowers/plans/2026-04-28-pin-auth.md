# Sprint 2 — PIN Auth + Elevation Gate (passkey-first, PIN-fallback)

**Branch:** `feat/pin-auth-sprint2`
**Spec author:** user (cv@ibba.group)
**Plan author:** Claude
**Date:** 2026-04-28

## Goal

Eliminar el bug donde usuarios Google OAuth no pueden agendar contactos (porque `gate-password` falla — no tienen contraseña). Reemplazar el password gate con un **elevation token** que se obtiene vía:
- **Passkey** si el usuario tiene una registrada (cero fricción con biometría),
- **PIN de 6 dígitos** como fallback universal.

Aplicar el elevation gate a las 6 operaciones sensibles definidas: **transferir, agregar contacto, cambio email, cambio PIN, cerrar cuenta, exportar datos**.

## Spec consolidada

| Decisión | Valor |
|---|---|
| Estrategia | Passkey-first + PIN-fallback (Opción C) |
| Formato PIN | 6 dígitos numéricos |
| Hashing | bcrypt cost 10 (Deno `https://deno.land/x/bcrypt@v0.4.1`) |
| Token de elevación | HMAC-SHA256 firmado, scope-bound, 60s TTL |
| Onboarding email/pwd | PIN en `Signup` step `set-pin` después de `verify-otp`, 2x para confirmar |
| Onboarding Google | PIN en `Signup` step `set-pin` después de `google-cuit-form`, 2x para confirmar |
| Recovery | Magic link de Supabase Auth → ruta `/pin-recovery` → reset con cooldown 24h |
| Lockout | 3 intentos → 15min, 5 intentos totales → bloqueo del usuario hasta intervención |
| Scope ops | transferir, agregar contacto, cambio email, cambio PIN, cerrar cuenta, exportar datos |

## Arquitectura

### Storage — `user_security` table

Separación limpia. RLS lockdown total. Solo edge functions con service_role pueden leer/escribir.

```sql
auth_user_id      UUID PK FK auth.users
pin_hash          TEXT      -- bcrypt(pin)
pin_set_at        TIMESTAMPTZ
failed_attempts   INT  DEFAULT 0
locked_until      TIMESTAMPTZ
total_lockouts    INT  DEFAULT 0
account_locked    BOOLEAN DEFAULT false
last_recovery_at  TIMESTAMPTZ  -- para cooldown 24h
created_at        TIMESTAMPTZ
updated_at        TIMESTAMPTZ
```

`pin_security_audit_log` con `event` (`set`, `verify_ok`, `verify_fail`, `lockout`, `recovery_requested`, `recovery_completed`, `change`).

### Token de elevación

Copia de `gateToken.ts` con campo extra `scope` para bind operación-específica.

```
Payload: { user_id, scope, exp_unix, jti }
TTL: 60s
HMAC-SHA256 con `GATE_TOKEN_SECRET` (reuso del existente)
```

`scope` examples: `transfer`, `add_contact`, `change_email`, `change_pin`, `close_account`, `export_data`.

### Edge Functions

| Function | Purpose |
|---|---|
| `pin-set-initial` | Set PIN durante onboarding. Falla si ya está seteado. |
| `pin-verify` | Verifica PIN, emite elevation token con scope. Aplica lockout. |
| `pin-change` | Cambia PIN (requiere PIN actual). |
| `pin-recovery-reset` | Reset PIN post-magic-link. Verifica que la sesión sea reciente y cumple cooldown 24h. |
| `pin-status` | Devuelve `{ pin_set, account_locked, locked_until }` para frontend. |
| `auth-elevate-passkey` | Verifica passkey y emite elevation token con scope. |

Existing `gate-password` se mantiene por backward-compat pero ya no se usa desde el frontend post-deploy.

### Frontend

- **Hook `useElevation()`** — ofrece `requestElevation(scope)` que: (1) chequea passkey → si hay, dispara passkey flow → token; (2) si no hay passkey o usuario cancela → muestra `<PinInputModal>` → token.
- **Component `<PinInputModal>`** — input 6 dígitos numéricos, soporte para link "Olvidé mi PIN" (dispara magic link), muestra estado lockout.
- **Component `<PinSetupForm>`** — 2 inputs (PIN + confirmación), validación de match, usado en onboarding y recovery.
- **Page `/pin-recovery`** — landing post-magic-link. Verifica auth fresca, llama `pin-recovery-reset`.
- **Page `/pin-setup` (forced)** — para usuarios existentes sin PIN seteado, redirect obligatorio post-login.

### Routing guard

Después de hidratar y cargar `cliente`, si `pin-status` devuelve `pin_set: false`, redirect a `/pin-setup`. Para nuevos usuarios el PIN se setea durante `Signup`, así que solo aplica a usuarios pre-Sprint-2.

## Tasks

### Task 1: Migration `user_security` + `pin_security_audit_log`

`supabase/migrations/20260428_user_security_and_pin.sql`

Tables, indexes, RLS deny-all (only service_role). Apply to remote DB via `supabase db query --linked -f`.

**Definition of Done:**
- [ ] File present in migrations dir
- [ ] Tables exist on remote with RLS enabled (`relrowsecurity=true`)
- [ ] No RLS policies allow direct user access (lockdown)

### Task 2: `pinCrypto.ts` (TDD)

bcrypt wrapper. Functions:
- `hashPin(pin: string): Promise<string>` — bcrypt cost 10
- `verifyPin(pin: string, hash: string): Promise<boolean>`
- Constant-time PIN format validation: `isValidPinFormat(s): boolean` (exactly 6 digits 0-9)

**Tests:** valid/invalid formats, hash round-trip, wrong PIN rejection, hash uniqueness (salt verification).

### Task 3: `elevationToken.ts` (TDD)

Copy of `gateToken.ts` with `scope` claim and `jti` for replay protection.

```typescript
signElevationToken(userId, scope, ttlSec, secret): Promise<string>
verifyElevationToken(token, expectedUserId, expectedScope, secret): Promise<boolean>
```

**Tests:** round-trip per scope, wrong scope rejection, tampered, expired, wrong user.

### Task 4: `pinLockout.ts` (TDD)

State machine for failed attempts.

```typescript
shouldLockout(failedAttempts: number, totalLockouts: number): { lock: boolean, until: Date | null, hardLock: boolean }
recordFailedAttempt(currentFailed, currentTotalLockouts): { newFailed, newLockedUntil, newTotalLockouts, accountLocked }
resetAttempts(): { failedAttempts: 0, lockedUntil: null }
```

Rules: 3 fails → 15min lock + reset failed counter, increment totalLockouts. 5 totalLockouts → `account_locked=true`.

**Tests:** boundary at 2/3 attempts, lockout expiry, hard-lock at 5th lockout.

### Task 5: Edge Function `pin-set-initial`

POST `/functions/v1/pin-set-initial` (Bearer JWT)
Body: `{ pin: string }`
Behavior: rejects if `pin_hash IS NOT NULL`. Hashes, inserts row in `user_security`. Audit log `event='set'`.

### Task 6: Edge Function `pin-verify`

POST `/functions/v1/pin-verify` (Bearer JWT)
Body: `{ pin, scope }`
Behavior: checks lockout state → verifies PIN → on success: reset attempts, emit elevation token; on fail: increment failed_attempts, possibly lock. Audit log per attempt.

Returns: `{ ok: true, elevation_token }` or `{ ok: false, reason: 'invalid_pin' | 'locked' | 'account_locked', locked_until?, attempts_remaining? }`.

### Task 7: Edge Function `pin-change`

POST `/functions/v1/pin-change` (Bearer JWT)
Body: `{ current_pin, new_pin }`
Behavior: verify current → hash new → update → audit log `event='change'`.

### Task 8: Edge Function `pin-recovery-reset`

POST `/functions/v1/pin-recovery-reset` (Bearer JWT, freshly authenticated via magic link)
Body: `{ new_pin }`
Behavior:
1. Verify session age (≤5min from `auth.users.last_sign_in_at` to ensure user came from magic link, not stale session)
2. Verify cooldown: `last_recovery_at IS NULL OR last_recovery_at < now() - interval '24h'`
3. Update PIN, reset attempts/lockouts, set `last_recovery_at = now()`
4. Audit log `event='recovery_completed'`

### Task 9: Edge Function `pin-status`

GET `/functions/v1/pin-status` (Bearer JWT)
Returns: `{ pin_set: boolean, account_locked: boolean, locked_until: ISO | null }`

### Task 10: Edge Function `auth-elevate-passkey`

POST `/functions/v1/auth-elevate-passkey` (Bearer JWT)
Body: `{ scope, passkey_response }` (SimpleWebAuthn assertion)
Behavior: verifies passkey via existing infra → emits elevation token with scope.

Coexiste con `passkey-auth-finish` (que es para login). Esta es para post-login elevation.

### Task 11: Client wrappers `pinApi.ts` + `elevationApi.ts`

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| Existing users have no PIN — first login post-deploy → forced `/pin-setup` | Acceptable. Show clear copy: "By security policy, set up your 6-digit PIN now." |
| bcrypt in Deno Edge Functions slow | Cost 10 + 6-digit input means ~50-80ms per verify. OK. |
| Magic link recovery via `resetPasswordForEmail` requires user has email | All users do (Google OAuth → email; email/pwd → email). |
| Account hard-lock (5 lockouts total) leaves user stuck | Document support@securepaynet email as recovery path. Add a `manual_unlock_request` admin flag for future. Out of scope this Sprint. |
| Replay of elevation token within 60s | `jti` field + Edge Function-side single-use enforcement. Defer if too complex; 60s + scope-binding is acceptable risk. |
